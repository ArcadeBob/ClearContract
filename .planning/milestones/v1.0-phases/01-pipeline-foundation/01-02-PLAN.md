**2. Analysis pass definitions:**
Define an array of passes. For Phase 1, implement exactly 3 passes to prove the engine works:

Pass 1 — "risk-overview":
- System prompt: Instructs Claude to analyze the contract from a glazing/glass installation subcontractor perspective, identifying the top risks across all categories. Focus on overall risk assessment, contract type identification, and client identification. Return findings for the most significant risks found.
- User prompt: "Analyze this glazing/glass installation contract. Identify the client name, contract type, and the most significant risks. Focus on indemnification, payment terms, insurance, scope clarity, and compliance requirements."

Pass 2 — "dates-deadlines":
- System prompt: Instructs Claude to extract all dates, deadlines, milestones, notice periods, cure periods, payment terms, and time-sensitive obligations from the contract.
- User prompt: "Extract all dates, deadlines, and time-sensitive terms from this contract. Include project start/completion dates, notice periods, cure periods, payment due dates, milestone dates, insurance certificate deadlines, and any other time-bound obligations."

Pass 3 — "scope-financial":
- System prompt: Instructs Claude to analyze scope of work definitions, financial terms, and potential scope gaps from a glazing subcontractor perspective.
- User prompt: "Analyze the scope of work and financial terms in this contract. Identify scope inclusions/exclusions, ambiguous scope language, retainage terms, payment schedules, change order procedures, and financial risk areas."

Each pass system prompt should:
- Specify the glazing/glass installation subcontractor perspective
- Instruct Claude to be thorough but precise — only report genuine findings
- Instruct Claude to include specific clause references where possible
- NOT ask for risk scores (those are computed deterministically)
- NOT ask for client name or contract type (except in risk-overview pass)

**3. PDF preparation function:**
```typescript
async function preparePdfForAnalysis(
  pdfBuffer: Buffer,
  fileName: string,
  client: Anthropic
): Promise<{ fileId: string; usedFallback: boolean }>
```

Logic:
- Get approximate page count from the PDF. Use a rough heuristic: count occurrences of "/Page" in the raw PDF buffer (each page object contains "/Type /Page"). This is a simple approximation — exact count not critical, just need to know if it's over ~100 pages.
- If estimated pages <= 100: Upload the raw PDF to Anthropic Files API using `client.beta.files.upload()` with the beta header. Return the file_id and `usedFallback: false`.
- If estimated pages > 100: Extract text using `unpdf`'s `extractText(new Uint8Array(pdfBuffer), { mergePages: true })`, then upload the extracted text as a `.txt` file to the Files API. Return the file_id and `usedFallback: true`.
- Wrap in try/catch. If native PDF upload fails for any reason, fall back to text extraction automatically (graceful degradation per user decision).

**4. Pass execution function:**
```typescript
async function runAnalysisPass(
  client: Anthropic,
  fileId: string,
  passName: string,
  systemPrompt: string,
  userPrompt: string
): Promise<{ passName: string; result: PassResult }>
```

Logic:
- Call `client.beta.messages.create()` with:
  - `model: 'claude-sonnet-4-20250514'`
  - `max_tokens: 8192`
  - `betas: BETAS`
  - `system: systemPrompt`
  - `messages`: single user message with two content blocks:
    1. Document block: `{ type: 'document', source: { type: 'file', file_id: fileId }, cache_control: { type: 'ephemeral' } }`
    2. Text block: `{ type: 'text', text: userPrompt }`
  - `output_config: { format: zodOutputFormat(PassResultSchema) }`

- Parse the response: `JSON.parse(response.content.filter(b => b.type === 'text').map(b => b.text).join(''))`
- Return `{ passName, result: parsedData }`

**COMPATIBILITY NOTE:** If `output_config` does not work with `client.beta.messages.create()` (the beta endpoint), try using the raw JSON schema approach instead of zodOutputFormat. The zodOutputFormat helper just converts the Zod schema to a JSON Schema object — you can extract it with `zodOutputFormat(PassResultSchema)` and pass it directly. Test during implementation and use whichever approach works.

function computeRiskScore(findings: Array<{ severity: string }>): number

**6. Result merging function:**
```typescript
function mergePassResults(
  results: PromiseSettledResult<{ passName: string; result: PassResult }>[],
  passes: Array<{ name: string }>

```

Logic:
- Iterate results. For fulfilled promises, collect findings and dates. For rejected promises, create an error finding (severity: Critical, category: Risk Assessment, title: "Analysis Pass Failed: {passName}", description: error message, recommendation: "Try uploading the contract again...").
- Build passResults array with status 'success' or 'failed' for each pass.
- Extract client name and contract type from the risk-overview pass result. The risk-overview pass needs a slightly different schema OR we extract client/contractType from the first pass differently.

The risk-overview pass needs to return client name and contract type in addition to findings/dates. Two approaches:
- Option A: Create a separate `RiskOverviewResultSchema` that extends PassResultSchema with `client` and `contractType` fields. Use this schema only for the risk-overview pass.
- Option B: Have the risk-overview pass put client/contractType info in a finding with a special title pattern that gets parsed out during merging.

**Use Option A** — create a `RiskOverviewResultSchema` in the schemas file (or inline it here) that adds `client: z.string()` and `contractType: z.enum([...])` to the base PassResultSchema. If importing from schemas file, note this needs to have been created in Plan 01. If it wasn't, define it inline in this file.

- Compute riskScore from all successful findings using `computeRiskScore()`.
- Deduplicate findings by title (if the same finding appears in multiple passes, keep the one with higher severity).
- Assign unique IDs to all findings: `f-${Date.now()}-${index}`.
- Default client to "Unknown Client" and contractType to "Subcontract" if risk-overview pass failed.

**7. Main handler function:**
```typescript
export default async function handler(req: VercelRequest, res: VercelResponse)
```

Logic:
- CORS handling: respond to OPTIONS with 200 and appropriate headers (Access-Control-Allow-Origin: *, methods: POST, OPTIONS, headers: Content-Type).
- Reject non-POST methods with 405.
- Validate API key exists (process.env.ANTHROPIC_API_KEY), return 500 if missing.
- Validate request body: `pdfBase64` must be a non-empty string. Return 400 if invalid.
- Decode base64 to Buffer. Check buffer size — if over 10MB (10 * 1024 * 1024 bytes), return 400 with "File too large" error. (10MB practical limit per research — the 25MB goal is deferred.)
- Initialize Anthropic client.
- Call `preparePdfForAnalysis()` to upload PDF and get fileId.
- Define the 3 passes array.
- Execute all passes in parallel with `Promise.allSettled()`.
- Call `mergePassResults()` to combine results.
- Clean up: delete the uploaded file from Files API in a finally block (best-effort, catch and ignore errors).
- Return 200 with the merged result JSON.
- Error handling: catch top-level errors. Check for 429 (rate limit), 401 (invalid key), and return appropriate HTTP status codes. For other errors, return 500 with generic message. Log errors with console.error.

**Key differences from old endpoint:**
- No pdf-parse import — uses native PDF via Files API or unpdf fallback
- No manual JSON parsing/fence stripping — structured outputs guarantee valid JSON
- No single 4096-token call — three parallel 8192-token passes
- No AI-determined risk score — deterministic computation from findings
- Partial results on failure — Promise.allSettled instead of all-or-nothing
