# Phase 43: Analysis Pipeline Server Writes - Research

**Researched:** 2026-03-18
**Domain:** Supabase server-side writes, JWT auth validation, Vercel serverless
**Confidence:** HIGH

## Summary

Phase 43 transforms the analysis pipeline from a stateless API (Claude analysis only, client stores results) into a server-owned write pipeline where the server authenticates the request, reads the company profile from Postgres, runs analysis, and writes all results (contract, findings, dates) directly to the database. The client simplifies to sending a PDF + auth token and receiving back the complete contract object.

The technical surface is well-understood: Supabase `createClient` with `SUPABASE_SERVICE_ROLE_KEY` for server writes (bypasses RLS), `auth.getUser(jwt)` for token validation, sequential inserts (contract -> findings -> dates), and `mapToSnake()`/`mapRow()` for data transformation. The client-side changes center on removing `companyProfile` from the request body, adding the `Authorization` header, and restructuring `handleUploadComplete` to eliminate the placeholder contract pattern.

**Primary recommendation:** Modify `api/analyze.ts` in-place -- add JWT validation at the top of the handler, create a service-role Supabase client, read profile, then after analysis completes insert rows and return the full contract with DB-assigned IDs.

## Architecture Patterns

### Server-Side Supabase Client (Recommended: Inline in analyze.ts)

Create the service-role client inside the request handler, not at module level. Vercel's warm starts can reuse module-level state across requests, which is fine for a stateless client but keeping it in-handler is cleaner for this single-use case.

```typescript
// Inside handler, after JWT validation
import { createClient } from '@supabase/supabase-js';

const supabaseAdmin = createClient(
  process.env.SUPABASE_URL!,
  process.env.SUPABASE_SERVICE_ROLE_KEY!,
  {
    auth: {
      autoRefreshToken: false,
      persistSession: false,
    },
  }
);
```

**Recommendation:** Keep inline in `analyze.ts` rather than creating a shared module. This is the only server endpoint that needs a service-role client. If Phase 44 (CRUD operations) needs server writes, extract then. YAGNI applies.

### JWT Validation Pattern

Extract token from Authorization header, validate with `auth.getUser(jwt)`:

```typescript
// At top of handler, after CORS/method checks
const authHeader = req.headers.authorization;
if (!authHeader?.startsWith('Bearer ')) {
  return res.status(401).json({ error: 'Authentication required' });
}
const token = authHeader.slice(7); // 'Bearer '.length

const { data: { user }, error: authError } = await supabaseAdmin.auth.getUser(token);
if (authError || !user) {
  return res.status(401).json({ error: 'Authentication required' });
}
const userId = user.id;
```

**Key detail:** `auth.getUser(jwt)` makes a network call to Supabase Auth to validate the token server-side. This catches expired/revoked tokens that local JWT parsing would miss. The trade-off is one additional network round-trip (~50-100ms), which is negligible against the 30-60s analysis time.

### Company Profile Read Pattern

```typescript
const { data: profileRow } = await supabaseAdmin
  .from('company_profiles')
  .select('*')
  .eq('user_id', userId)
  .maybeSingle();

const companyProfile: CompanyProfile = profileRow
  ? { ...DEFAULT_COMPANY_PROFILE, ...mapRow<CompanyProfile>(profileRow) }
  : DEFAULT_COMPANY_PROFILE;
```

**Note:** Use `.maybeSingle()` not `.single()` -- the latter throws when no row exists. The `DEFAULT_COMPANY_PROFILE` spread ensures all fields are populated even if the DB row has defaults/empty strings.

### Sequential Insert Pattern

After analysis completes, write results to DB in order: contract first (to get the ID), then findings and dates in bulk:

```typescript
// 1. Insert contract row
const contractPayload = mapToSnake({
  userId,
  name: fileName.replace(/\.pdf$/i, ''),
  client: merged.client,
  type: merged.contractType,
  uploadDate: new Date().toISOString().split('T')[0],
  status: 'Reviewed',
  riskScore: scoreResult.score,
  scoreBreakdown: merged.scoreBreakdown,  // JSONB -- stored as camelCase
  bidSignal,                               // JSONB -- stored as camelCase
  passResults: merged.passResults,         // JSONB -- stored as camelCase
});
// Remove undefined values that mapToSnake may produce
delete contractPayload.id;  // Let Postgres generate

const { data: contractRow, error: contractError } = await supabaseAdmin
  .from('contracts')
  .insert(contractPayload)
  .select()
  .single();

if (contractError) throw new Error(`Failed to save contract: ${contractError.message}`);

const contractId = contractRow.id;

// 2. Bulk insert findings
const findingPayloads = deduplicatedFindings.map((f) => {
  const payload = mapToSnake({
    contractId,
    userId,
    severity: f.severity,
    category: f.category,
    title: f.title,
    description: f.description,
    recommendation: f.recommendation,
    clauseReference: f.clauseReference || 'N/A',
    negotiationPosition: f.negotiationPosition || '',
    actionPriority: f.actionPriority || 'monitor',
    resolved: false,
    note: '',
    clauseText: f.clauseText,
    explanation: f.explanation,
    crossReferences: f.crossReferences,   // JSONB
    legalMeta: f.legalMeta,               // JSONB
    scopeMeta: f.scopeMeta,               // JSONB
    sourcePass: f.sourcePass,
    downgradedFrom: f.downgradedFrom,
    isSynthesis: f.isSynthesis,
  });
  delete payload.id;
  return payload;
});

const { data: findingRows, error: findingsError } = await supabaseAdmin
  .from('findings')
  .insert(findingPayloads)
  .select();

// 3. Bulk insert dates
const datePayloads = merged.dates.map((d) => mapToSnake({
  contractId,
  userId,
  label: d.label,
  date: d.date,
  type: d.type,
}));

const { data: dateRows, error: datesError } = await supabaseAdmin
  .from('contract_dates')
  .insert(datePayloads)
  .select();
```

### Response Shape Change

The server must return a complete `Contract` object with DB-assigned IDs that the client can directly `addContract()` with:

```typescript
// Build response matching Contract type
const contract = {
  ...mapRow(contractRow),

  dates: dateRows ? mapRows(dateRows) : [],

return res.status(200).json(contract);
```

### Client-Side Changes

**`analyzeContract.ts`:**
- Accept `accessToken: string` parameter
- Remove `loadCompanyProfile()` import and call
- Add `Authorization: Bearer ${token}` header
- Simplify body to `{ pdfBase64, fileName }`
- Update response validation to expect a `Contract` shape (not `AnalysisResult`)

**`App.tsx` - `handleUploadComplete`:**

- No more placeholder contract creation
- Set `analyzingId` to a sentinel (e.g., `'pending'`) for UI state
- On success: `addContract(result)` then `navigateTo('review', result.id)`
- On failure: toast with retry, no cleanup needed (no placeholder to delete)

**`App.tsx` - `handleReanalyze`:**
- Same auth token pattern
- Server creates new contract row (or could update -- but context says "same pattern")
- Finding preservation logic applies client-side to the returned findings before calling `addContract()`

### Anti-Patterns to Avoid
- **Module-level Supabase client with user token:** The service-role client should NOT use user tokens. It bypasses RLS entirely and sets `user_id` explicitly.
- **Trusting client-side user_id:** Never accept `user_id` from the request body. Always extract from the validated JWT.
- **Using `.single()` for profile lookup:** Use `.maybeSingle()` since the row may not exist for new users.

## Common Pitfalls

### Pitfall 1: JSONB Columns Need camelCase, Not snake_case
**What goes wrong:** `mapToSnake()` converts top-level keys but JSONB values (scoreBreakdown, bidSignal, passResults, legalMeta, scopeMeta, crossReferences) must stay as camelCase objects since that is how they are stored and read.
**Why it happens:** `mapToSnake()` only converts top-level keys (by design). The JSONB values pass through unchanged.
**How to avoid:** This actually works correctly by default. Just ensure JSONB values are assigned directly from the analysis result objects (which are already camelCase). Do NOT recursively convert JSONB contents.
**Warning signs:** If `legalMeta.clauseType` becomes `legal_meta.clause_type` in the DB, something is wrong.

### Pitfall 2: Missing Required Fields with NOT NULL Constraints
**What goes wrong:** Insert fails with "null value in column X violates not-null constraint."
**Why it happens:** Analysis results may have optional fields (`negotiationPosition`, `actionPriority`) that the DB requires as NOT NULL.
**How to avoid:** Provide defaults for all NOT NULL fields: `negotiationPosition: f.negotiationPosition || ''`, `actionPriority: f.actionPriority || 'monitor'`, `clauseReference: f.clauseReference || 'N/A'`, `note: ''`, `resolved: false`.
**Warning signs:** Findings from synthesis pass or failed-pass findings lack some fields.

### Pitfall 3: CORS Header Must Include Authorization
**What goes wrong:** Browser blocks the request with a CORS preflight failure because `Authorization` is not in `Access-Control-Allow-Headers`.
**Why it happens:** The current handler only allows `Content-Type`.
**How to avoid:** Update the CORS header: `res.setHeader('Access-Control-Allow-Headers', 'Content-Type, Authorization');`
**Warning signs:** Works in Postman/curl but fails in browser.

### Pitfall 4: Environment Variables Missing on Server
**What goes wrong:** `SUPABASE_URL` and `SUPABASE_SERVICE_ROLE_KEY` are undefined in the serverless function.
**Why it happens:** Client uses `VITE_SUPABASE_URL` (Vite prefix), but server code uses `process.env.SUPABASE_URL` (no prefix). These are different env vars.
**How to avoid:** Ensure both `SUPABASE_URL` and `SUPABASE_SERVICE_ROLE_KEY` are set in `.env.local` (for local dev) and Vercel project settings (for production). The `VITE_` prefix versions are client-only.
**Warning signs:** Works locally with `vercel dev` (which reads `.env.local`) but fails in production.

### Pitfall 5: Removing Client-Side ID Generation Too Early
**What goes wrong:** The client tries to use `f-${crypto.randomUUID()}` for finding IDs but the server now generates them.
**Why it happens:** The existing `analyze.ts` handler generates client-style IDs (`f-${crypto.randomUUID()}`). These must be removed since Postgres now generates UUIDs.
**How to avoid:** Remove the `findingsWithIds` mapping in `analyze.ts`. The DB generates IDs via `gen_random_uuid()`. The `.select()` after `.insert()` returns the generated IDs.

### Pitfall 6: Re-analyze Creates Duplicate Contract
**What goes wrong:** Re-analyze creates a new contract row instead of updating the existing one.
**Why it happens:** The context says "follows the same pattern" for re-analyze, which could mean a new insert.
**How to avoid:** For re-analyze, the server should receive the existing `contractId` and update that row (plus delete old findings/dates and insert new ones). The CASCADE delete on findings/dates is available but a targeted approach is cleaner: delete old findings + dates, then insert new ones.
**Warning signs:** After re-analyze, two contract entries appear in the contracts list.

```typescript
export async function analyzeContract(file: File, accessToken: string): Promise<Contract> {
  // ... file validation ...
  const pdfBase64 = await readFileAsBase64(file);

  const response = await fetch('/api/analyze', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'Authorization': `Bearer ${accessToken}`,
    },
    body: JSON.stringify({ pdfBase64, fileName: file.name }),
  });

  // ... error handling ...
  const json = await response.json();
  // Response is now a full Contract object with DB IDs
  return json;
}
```

### Getting Session Token in App.tsx
```typescript
const handleUploadComplete = async (file: File) => {
  const { data: { session } } = await supabase.auth.getSession();
  if (!session?.access_token) {
    showToast({ type: 'error', message: 'Please sign in to analyze contracts.' });
    return;
  }

  setAnalyzingId('pending');

  try {
    const contract = await analyzeContract(file, session.access_token);
    addContract(contract);
    navigateTo('review', contract.id);
  } catch (err) {
    const classified = classifyError(err);
    showToast({
      type: 'error',
      message: classified.userMessage,
      ...(classified.retryable ? { onRetry: () => handleUploadComplete(file) } : {}),
    });
  } finally {
    setAnalyzingId(null);
  }
};
```

## Open Questions

1. **Re-analyze: update vs new contract?**
   - What we know: Context says "follows the same pattern" suggesting a new insert, but also mentions "update the existing contract row instead of inserting a new one"
   - What's unclear: Whether re-analyze should UPDATE the existing contract or INSERT a new one and delete the old
   - Recommendation: UPDATE existing contract row + DELETE old findings/dates + INSERT new ones. This preserves the contract ID for bookmarks/history. The server endpoint can accept an optional `contractId` field in the request body to signal re-analyze mode.

2. **Response schema validation on client**
   - What we know: Currently uses `AnalysisResultSchema` to validate API response
   - What's unclear: The response shape changes from `AnalysisResult` to `Contract` (with nested findings/dates containing DB IDs)
   - Recommendation: Either update the existing schema or create a new one. The response will have `id` (UUID), `userId`, `createdAt`, `updatedAt` fields that `AnalysisResult` does not expect. Simplest approach: trust the server response (it is our own server), or create a lightweight `ContractResponseSchema`.

3. **Partial write error handling detail**
   - What we know: Decision says "contract exists but incomplete is acceptable"
   - What's unclear: Should the server return an error (500) or a success with a warning flag?
   - Recommendation: If contract insert succeeds but findings/dates fail, log the error, return the contract with empty findings/dates arrays, and let the user re-analyze. Return 200 (not 500) since the contract was created.
