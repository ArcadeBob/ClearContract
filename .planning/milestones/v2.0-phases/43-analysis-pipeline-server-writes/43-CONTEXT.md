# Phase 43: Analysis Pipeline Server Writes - Context

**Gathered:** 2026-03-18
**Status:** Ready for planning

<domain>
## Phase Boundary

The analysis pipeline authenticates requests via JWT, reads the company profile from Supabase (not localStorage), and writes all results (contract, findings, dates) directly to Postgres. The server owns contract creation — no client-side placeholder row in the database. The client sends the auth token and PDF; the server does everything else.

</domain>

<decisions>
## Implementation Decisions

### Auth token flow
- Client sends JWT via `Authorization: Bearer <token>` header (standard pattern)
- Client gets token from `supabase.auth.getSession()` before calling `/api/analyze`
- Server validates with `supabase.auth.getUser(token)` — authoritative, handles expired/revoked tokens
- Unauthenticated requests get 401 with generic `{ error: 'Authentication required' }` — no leak of missing/expired/invalid distinction (matches Phase 40 security pattern)
- CORS `Access-Control-Allow-Headers` updated to include `Authorization` alongside `Content-Type`

### Upload UX transition
- Upload page shows analysis-in-progress state (spinner/animation) — no navigation until complete or user navigates away
- User CAN navigate away during analysis — analysis continues in background
- On success (user still on upload page): navigate to review page with the new contract
- On success (user navigated away): success toast with clickable "View" action that navigates to the review page
- On failure: error toast with retry button (existing `classifyError` pattern), stay on current page
- No client-side placeholder contract in-memory during analysis — the contract only exists after server returns it
- `handleUploadComplete` in App.tsx changes significantly: no more `addContract(placeholder)` + `navigateTo('review')` upfront

### Server write strategy
- Use `SUPABASE_SERVICE_ROLE_KEY` to create a server-side Supabase client that bypasses RLS
- Set `user_id` column explicitly from the validated JWT's user ID on every insert
- Sequential inserts: contract row first, then bulk-insert findings, then bulk-insert dates
- If findings/dates insert fails after contract insert, contract exists but incomplete — user can re-analyze (acceptable tradeoff over Postgres function complexity)
- Contract ID generated by Postgres `gen_random_uuid()` default — read back from insert response
- Finding IDs also generated by Postgres (no more client-side `f-${crypto.randomUUID()}`)
- Server returns full contract object with findings and dates (same shape as current API response but with DB-assigned IDs)
- Client calls `addContract()` to put the returned contract into the in-memory store
- `mapToSnake()` from `src/lib/mappers.ts` used for camelCase→snake_case write payloads

### Profile read on server
- Server reads company profile from `company_profiles` table using the authenticated user_id
- Server-side Supabase client (service role) queries the profile — no RLS needed
- If no profile row exists (user never visited Settings): fall back to `DEFAULT_COMPANY_PROFILE` constant
- `DEFAULT_COMPANY_PROFILE` imported from `src/knowledge/types.ts` on the server side
- Client-side `analyzeContract.ts` removes `loadCompanyProfile()` import and `companyProfile` from request body
- Request body simplified to `{ pdfBase64, fileName }` only
- Server-side `CompanyProfileSchema` validation on the request body is removed (profile no longer in body)
- `AnalyzeRequestSchema` updated to remove `companyProfile` field

### Claude's Discretion
- Exact server-side Supabase client initialization pattern (inline vs module)
- Whether to create a shared server Supabase utility or keep it in analyze.ts
- Exact upload page progress UI during analysis (reuse existing ContractUpload analyzing state)
- Whether `handleUploadComplete` refactor splits into separate functions or stays inline
- Error message wording for partial write failures (contract exists but findings failed)

</decisions>

## Existing Code Insights

### Reusable Assets
- `mapToSnake()` (`src/lib/mappers.ts`): Convert camelCase analysis results to snake_case for Postgres inserts
- `mapRow()` (`src/lib/mappers.ts`): Convert inserted rows back to camelCase for the response
- `classifyError()` (`src/utils/errors.ts`): Error classification for client-side error handling
- `computeBidSignal()` (`src/utils/bidSignal.ts`): Already called in analyze.ts — stays as-is
- `DEFAULT_COMPANY_PROFILE` (`src/knowledge/types.ts`): Server-side fallback when no profile row exists
- `useToast` hook: For success/error notifications on the client side
- `ContractUpload` component: Already has `isAnalyzing` and `onCancel` props — may need enhancement for background analysis

### Established Patterns
- Server function uses `@vercel/node` types (VercelRequest, VercelResponse)
- Zod validation on request body (AnalyzeRequestSchema)
- Custom undici fetch for Anthropic API calls (timeout configuration)
- Error classification with `classifyError()` on both client and server
- In-memory store mutations via `addContract()`, `updateContract()` in useContractStore

### Integration Points
- `analyzeContract()` in `src/api/analyzeContract.ts` is called from `App.tsx` `handleUploadComplete` and `handleReanalyze`
- `handleUploadComplete` creates placeholder + navigates + runs analysis — this entire flow changes
- `handleReanalyze` also calls `analyzeContract()` — needs auth token too
- Toast system for completion/failure notifications
- Router `navigateTo('review', contractId)` for post-analysis navigation

</code_context>
