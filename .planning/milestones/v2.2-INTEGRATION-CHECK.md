Complete. Traced path:

1. ContractUpload triggers handleUploadComplete(file) in App.tsx
2. App.tsx fetches Supabase session and calls analyzeContract(file, accessToken) from src/api/analyzeContract.ts
3. analyzeContract base64-encodes the PDF and POSTs to /api/analyze with Authorization header
4. api/analyze.ts runs two-stage pipeline: primer pass (risk-overview) alone first, then 15 parallel passes, each with a 90s AbortController timeout; global 250s safety timeout fires isGlobalTimeout = true and aborts all controllers

8. User navigated to review page via navigateTo with contract.id

## lifecycle_status vs status Independence

Confirmed fully independent:

- Contract.status (Analyzing | Reviewed | Partial | Draft) -- analysis pipeline state, written by server only, never modified by user interaction

## Detailed Findings

### Finding 1: Re-analyze on a Partial Contract -- Edge Case

Severity: Low -- informational, not a blocking defect.

When a user re-analyzes a Partial contract and the re-analysis also hits the global timeout, api/analyze.ts performs an UPDATE on the existing contract row (lines 649-655) and deletes old analysis_usage rows (line 660) before inserting new ones. If the global timeout fires before the DB UPDATE itself completes (an extreme edge case), the finally block still cleans up the uploaded file and closes the undici Agent, but no contract update is persisted. The handler returns a 500, and handleReanalyze in App.tsx rolls back in-memory state to the pre-reanalysis snapshot. Behavior is safe.

No affected requirement.

### Finding 2: CostSummaryBar on Partial Contracts -- Confirmed Working

Severity: None -- correct behavior confirmed.

A Partial contract has analysis_usage rows for all passes that completed before the global timeout. useAnalysisUsage queries by contract_id and CostSummaryBar renders whatever rows exist. The component returns null only when rows.length === 0. In practice a Partial contract always has at least the primer-pass usage row, because a failed primer causes a thrown error (not a Partial save).

No affected requirement.

### Finding 3: useAnalysisUsage RLS -- Client vs. Service Role

Severity: Low -- by design, not a defect.

analysis_usage writes in api/analyze.ts use supabaseAdmin (service role, bypasses RLS). Client reads via useAnalysisUsage use the authenticated Supabase client with RLS requiring auth.uid() = user_id. If the session expires between upload completion and CostSummaryBar load, the query returns 0 rows and CostSummaryBar renders null silently -- acceptable degradation.

Affects: COST-03. No remediation needed; behavior is safe.
