## Phase Details

### Phase 11: Data Persistence
**Goal**: Contracts survive page refresh -- all analyzed contract data persists in localStorage and loads on app start
**Depends on**: None (v1.1 complete)
**Requirements**: PERSIST-01, PERSIST-02, PERSIST-03, PERSIST-04
**Success Criteria** (what must be TRUE):
  1. Upload a contract, refresh the page, and the contract with all findings/dates/metadata is still there
  2. On first visit (empty localStorage), mock contracts load as seed data; on subsequent visits, only persisted data loads
  3. If localStorage is full, user sees a warning and the app continues working (in-memory fallback)
  4. All contract fields serialize correctly: findings with legalMeta/scopeMeta, dates, bidSignal, passResults

### Phase 12: Contract Management
**Goal**: Users can delete contracts they no longer need, with proper confirmation and state cleanup
**Depends on**: Phase 11 (persistence must exist before delete is meaningful)
**Requirements**: MGMT-01, MGMT-02, MGMT-03
**Plans:** 1 plan

Plans:
- [ ] 12-01-PLAN.md -- Add delete operation with confirmation dialog and navigation guard

**Success Criteria** (what must be TRUE):
  1. Each contract card/review page has a delete button that triggers a confirmation dialog
  2. Confirmed deletion removes the contract from both React state and localStorage
  3. If the currently-viewed contract is deleted, user is navigated to dashboard

### Phase 13: Upload Error Feedback
**Goal**: Users see clear, actionable error messages when uploads fail -- no silent failures
**Depends on**: None (independent of persistence)
**Requirements**: UX-01, UX-02, UX-03, UX-04
**Success Criteria** (what must be TRUE):
  1. Dropping a .docx file shows "Only PDF files are accepted" inline in the upload zone
  2. Dropping a 15MB PDF shows "File exceeds 10MB limit (15MB)" inline in the upload zone
  3. API analysis failure shows a dismissible toast/banner at the top of the page (not hidden inside findings)
  4. Network timeout shows "Connection failed. Check your internet and try again." with a retry button

### Phase 14: Empty States and Dashboard Polish
**Goal**: Every view handles zero-data gracefully with helpful CTAs, and the dashboard shows only real data
**Depends on**: Phase 11 (empty states matter most when persistence means users start fresh)
**Requirements**: DASH-01, DASH-02, DASH-03, DASH-04
**Success Criteria** (what must be TRUE):
  1. Dashboard with 0 contracts shows a welcoming illustration/message and "Upload Your First Contract" CTA
  2. DateTimeline with 0 dates shows "No dates found in this contract"
  3. Dashboard stat cards show actual values only -- trend percentages removed (no fake data)
  4. ContractCard shows the risk score number (e.g., "Risk: 78/100") alongside finding counts
