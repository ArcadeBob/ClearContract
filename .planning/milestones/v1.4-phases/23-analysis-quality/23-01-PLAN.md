   - This is needed by the staleness check to inspect all modules

2. **Add checkModuleStaleness function** in `api/merge.ts`:
   - Import `getAllModules` from knowledge registry
   - Function signature: `function checkModuleStaleness(): UnifiedFinding[]`
   - For each module where `new Date(module.expirationDate) < new Date()`:
     * Create a UnifiedFinding with:
       - severity: 'Info'
       - category: 'Risk Assessment'
       - title: `Knowledge Module Outdated: ${module.title}`
       - description: `The ${module.title} module (effective ${module.effectiveDate}) expired on ${module.expirationDate}. Findings derived from this module should be verified against current statutes.`
       - recommendation: 'Verify findings from this module against current law before relying on them.'
       - clauseReference: 'N/A'
       - sourcePass: 'staleness-check'
   - Return the array of staleness findings (empty if all modules are current)

3. **Integrate staleness check into mergePassResults**:
   - After the deduplication phase and before computing risk score, call `checkModuleStaleness()`
   - Append staleness findings to `deduplicatedFindings` array
   - Staleness findings have severity 'Info' (weight 0) so they do not affect risk score
   - They will appear in the findings list as informational notices

Note: The staleness check uses `new Date()` at analysis time. During development all expirationDate values are set to future dates, so no staleness findings will appear unless a module is deliberately expired for testing.
