**DEBT-03 -- Zod-typed merge results:**
In `api/merge.ts`, replace the `Record<string, unknown>` parameter types in `buildBaseFinding`, `convertLegalFinding`, and `convertScopeFinding` with proper Zod-inferred types.

Approach: Import the per-pass finding schemas from the schema files. The findings from each pass are already typed as `z.infer<typeof PassResultSchema>['findings'][number]` which is the FindingSchema inferred type. Since all findings share the base FindingSchema shape, create a type alias:
```typescript
import type { FindingResult } from '../src/schemas/analysis';
```
If `FindingResult` is not exported, add `export type FindingResult = z.infer<typeof FindingSchema>;` to `src/schemas/analysis.ts`.

Then in merge.ts:
1. Change `buildBaseFinding(finding: Record<string, unknown>, ...)` to `buildBaseFinding(finding: FindingResult & Record<string, unknown>, ...)`. The intersection allows accessing both the known base fields without casts AND the pass-specific extension fields that vary per pass.
2. For the base fields (severity, category, title, description, recommendation, clauseReference, clauseText, explanation, negotiationPosition, downgradedFrom), access directly without `as` casts since FindingResult types them.
3. For pass-specific fields (riskType, hasInsuranceGap, paymentType, etc.), keep the `as` casts on the `Record<string, unknown>` side -- these fields come from pass-specific schemas and are not on the base FindingResult. This is the pragmatic approach described in RESEARCH.md.
4. Remove `as unknown as Record<string, unknown>` from the call sites in `mergePassResults` (lines 251, 256, 265). Instead, cast findings to `FindingResult & Record<string, unknown>` which is more precise.
