**Create `src/schemas/analysis.ts`** — the Zod schemas used by structured outputs in every API call.

Define the following schemas:

1. `FindingSchema` — z.object matching the Finding type with these fields:
   - `severity`: z.enum(['Critical', 'High', 'Medium', 'Low', 'Info'])
   - `category`: z.enum(['Legal Issues', 'Scope of Work', 'Contract Compliance', 'Labor Compliance', 'Insurance Requirements', 'Important Dates', 'Financial Terms', 'Technical Standards', 'Risk Assessment'])
   - `title`: z.string() — short descriptive title
   - `description`: z.string() — detailed explanation
   - `recommendation`: z.string() — action to take
   - `clauseReference`: z.string() — section/clause reference
   - `clauseText`: z.string().optional() — exact verbatim clause text (for future phases)
   - `explanation`: z.string().optional() — plain-English explanation of why problematic (for future phases)

   IMPORTANT: Do NOT use .min()/.max()/.minLength()/.maxLength() constraints — structured outputs does not support them. Keep schemas simple with basic types, enums, arrays, and nested objects only.

2. `ContractDateSchema` — z.object:
   - `label`: z.string()
   - `date`: z.string()
   - `type`: z.enum(['Start', 'Milestone', 'Deadline', 'Expiry'])

3. `PassResultSchema` — the schema each analysis pass returns:
   - `findings`: z.array(FindingSchema)
   - `dates`: z.array(ContractDateSchema)

4. `MergedAnalysisResultSchema` — the final merged response sent to the client:
   - `client`: z.string()
   - `contractType`: z.enum(['Prime Contract', 'Subcontract', 'Purchase Order', 'Change Order'])
   - `riskScore`: z.number()
   - `findings`: z.array(FindingSchema)
   - `dates`: z.array(ContractDateSchema)
   - `passResults`: z.array(z.object({ passName: z.string(), status: z.enum(['success', 'failed']), error: z.string().optional() }))

Export all schemas. Also export TypeScript types inferred from schemas using `z.infer<typeof SchemaName>` for use in the server code.

**Extend `src/types/contract.ts`** — add optional fields to the Finding interface for future phases:

Add to the Finding interface:
- `clauseText?: string` — exact verbatim clause text from the contract
- `explanation?: string` — plain-English explanation of why it's problematic

These are optional so existing code (FindingCard, etc.) continues to work without changes.

Also add a `passResults` field to the Contract interface:
- `passResults?: Array<{ passName: string; status: 'success' | 'failed'; error?: string }>` — tracks which analysis passes succeeded/failed

Keep all existing fields and types exactly as they are. The 9 categories remain unchanged. The severity levels remain unchanged. This is purely additive.
